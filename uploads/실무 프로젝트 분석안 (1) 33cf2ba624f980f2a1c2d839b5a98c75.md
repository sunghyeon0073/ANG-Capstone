# 4️⃣ 사이트 주요 정책 정의

## 1. 회원 정책

### 1.1 회원 유형 및 가입 조건

ANG는 조직 단위(부서)로 운영되며, 가입 시 부서 고유 코드를 필수 입력합니다.

| 회원 유형 | 설명 | 가입 방식 | 기능코드 |
| --- | --- | --- | --- |
| 일반 사용자 | 업무 실무자, 팀원 | 사번 + 부서코드 자가 가입 후 관리자 승인 | Auth-03 |
| 관리자 | 팀장 | 별도 관리자 계정 부여 | Admin-04 |
| 최고 관리자 | 시스템 전체 운영자 | 사전 등록 (시스템 설정) | Admin-04 |

### 1.2 가입 절차

기능정의서 Auth-03 기준 — 필수 입력 항목: 이름, 사번, 생년월일, 이메일, 비밀번호, 비밀번호 확인, 부서 고유 코드

- 가입 후 관리자 승인 대기 상태로 전환 (플로우차트: 관리자 승인 완료 후 로그인 가능)
- 관리자가 미승인 시 서비스 이용 불가
- 고유 코드 : 부서 별 고유코드를 가지고 있음

### 1.3 로그인 및 인증 정책

- 로그인 방식: 사번 + 비밀번호 조합 (Auth-01)
- 비밀번호 조건: 6자리이상,  25자리미만 , 영문 + 특수문자 조합 필수
- 비밀번호 변경: 마이페이지에서 직접 변경 가능 (User-01-3)
- 세션 만료 시 자동 로그아웃, 재로그인 필요

### 1.4 회원 정보 관리

- 개인정보 수정 항목 : 사진, 이메일, 연락처,생년월일, 이름 (User-01-2)
- 사번, 직급 등 기관 고유 정보는 관리자 및 최고 관리자 권한으로만 변경 가능 (Teamjang-01)
- 회원 탈퇴(삭제)는 관리자 및 최고 관리자만 처리 가능 (Admin-03)
- 탈퇴 시 해당 회원이 작성한 문서, 게시글, 업무 데이터, 메일 기록, 파일, 전자 결재 내역은 익명화 처리 후 조직 자산으로 보존
- 탈퇴 시 해당 회원의 채팅 기록, 개인 캘린더 기록 삭제

비밀번호 암호화 - 표준화 찾아보기

## 2. 디자인 정책

### 2.1 UI/UX 기본 원칙

- 1주일 내 적응 목표 — 즉시 사용 가능한 인터페이스 제공
- 핵심 기능( 캘린더, 문서 뷰어 )은 단일 웹 인터페이스에서 접근 가능
- 브라우저 기반 서비스 — 별도 클라이언트 설치 없이 사용
- 다크 모드, 라이트 모드 — 사용자 취향 고려 모드 선택 가능

### 2.2 화면 구성 기준

| 구분 | 구성 원칙 | 비고 |
| --- | --- | --- |
| 구분 | 구성 원칙 | 비고 |
| 메인 화면 | 알람 센터, 서비스 선택, 조직도 접근을 한 화면에서 제공 | 플로우차트 메인화면 기준 |
| 업무 보드 | 대기 / 진행 중 / 완료 칸반 방식 | 팀원별 업무 현황 한눈에 확인 |
| 캘린더 | 개인 캘린더 + 공유 캘린더 분리 표시, AI 일정 추천 버튼 노출 | AI-02-1 연동 |
| 파일함 | 개인 파일함 / 부서 공유 파일함 탭 분리 | File-01-4 기준 |
| AI 문서 | 문서 추천 리스트 → 양식 선택 → 초안 작성 → 수정 → 저장 순서로 단계별 화면 구성 | AI-01-1~5 기준 |
| OCR 변환 | 이미지 업로드 → 텍스트 반환 → 수정 여부 선택 → 저장 단계 구분 표시 | AI-01-5 기준 |

| 조직도 | 부서 전체 구성원 열람, 직원 클릭 시 상세 프로필 확인 및 채팅/메일 바로가기 | Common-01 기준 |
| 채팅 | 1:1 채팅 / 그룹 채팅 유형 선택 화면 분리, 메시지 작성 및 파일 첨부 | Chat-01-1/2 기준 |
| 메일함 | 받은 메일 전체 조회, 메일 작성(송수신자/제목/내용/첨부), 메일 삭제 | Mail-01~04 기준 |
| 게시판 | 공지사항 / 자유게시판 탭 분리, 통합 검색 및 필터(확장자/카테고리/기간) 상단 고정 | Common-04, Search-01 기준 |
| 알람 센터 | 결재 / 채팅 / 메일 / 업무 추천 알림 유형별 탭 분리, 전체 조회 기본 화면 | Alert-01 기준 |
| 마이페이지 | 내 정보 조회/수정, 비밀번호 변경 | User-01-1~3 기준 |
| 관리자 화면 | 회원 정보 조회/삭제, 계정 권한 관리, 직급 수정 — 일반 사용자 화면과 완전 분리 | Admin-01~05, Teamjang-01 기준 |

### 2.3 AI 기능 UI 노출 정책

- AI 추천 결과는 '추천' 레이블을 명시하여 AI 생성 콘텐츠임을 사용자에게 고지
- AI 문서 초안은 미리보기 후 사용자 확인 단계를 거쳐 저장 (AI-01-4)
- OCR 변환 결과는 수정 여부를 사용자가 선택 가능 (플로우차트 기준)

| 권한 등급 | 대상 | 주요 가능 기능 |

| 일반 사용자 | 팀원, 실무자 | 개인 캘린더 CRUD, 개인 파일함, 채팅, 메일, 게시판 조회/작성, AI 기능 |
| 부서 관리자(팀장) | 팀장급 | 공유 캘린더 CRUD, 부서 파일함 관리, 직급 수정, 공지사항 관리, 전자결재 승인/반려, 결재 대리인 지정 |
| 최고 관리자 | 시스템 운영자 | 회원 정보 조회/삭제, 계정 권한 관리, 전체 회원 관리 |

### 3.3 전자결재 운영 기준

- 결재 구조는 단일 결재자(부서 관리자) 기준으로 운영하며, 다단계 결재선은 현재 버전에서 지원하지 않음
- 부서 관리자 부재(휴가, 출장, 퇴직 등) 시 임시 결재 대리인을 지정할 수 있음
- 대리인 지정은 부서 관리자 본인 또는 최고 관리자만 가능 (Approval-01-5)
- 대리인 지정 기간은 명시적으로 설정하며, 기간 만료 시 자동 해제
- 대리인이 처리한 결재 이력에는 '대리 결재' 표시를 명시하여 원 결재자와 구분

### 3.4 파일 접근 권한

- 개인 파일함: 본인만 접근 가능, 타인 열람 불가
- 부서 공유 파일함: 같은 부서 구성원만 접근, 타 부서 차단
- 파일 수정/삭제 권한은 업로드한 본인 또는 부서 관리자에 한함 (플로우차트 '파일 수정/삭제 권한이 있는가?' 분기 기준)
- 파일 내보내기(PDF, DOCX)는 해당 파일 접근 권한 보유 시 가능 (File-01-7)

### 3.5 로컬 AI 서버 접근

- ANG AI 서버는 외부 인터넷과 차단된 로컬 서버로 운영 (Ollama 기반)
- 조직 내부 네트워크에서만 서비스 접근 가능 — 외부 IP 접근 원칙적 차단
- 부서별 독립 컨테이너 환경 분리 (Docker Compose 기준)

## 4. 콘텐츠 정책

### 4.1 문서 콘텐츠

- 업로드 가능 파일 형식: HWP, PDF, Excel, 이미지 파일 (File-01-1)
- 파일 내보내기 지원 형식: PDF, DOCX (File-01-7)
- OCR 변환(AI-01-5): 이미지/스캔 문서 → 텍스트 변환 후 사용자 확인 필수
- AI 문서 작성(AI-01-2)은 조직 내 기존 문서 히스토리 기반 생성 — 외부 데이터 미사용
- AI 생성 문서는 사용자 검토 및 수정 후 저장 원칙 (AI-01-3, 플로우차트 '추가 작업 필요?' 분기)

파일크기, 갯수

### 4.2 메일 및 채팅 콘텐츠

- 메일 작성 항목: 송수신자, 제목, 내용, 파일 첨부(선택) (Mail-01)
- 채팅 유형: 1:1 개인 채팅 (Chat-01-1), 그룹 채팅 1:M (Chat-01-2)
- 메일 및 채팅은 조직 내부 구성원 간 통신에 한정
- 부적절한 언어, 개인정보 포함 내용, 업무 외 콘텐츠의 과도한 사용은 관리자 모니터링 대상

### 4.3 AI 콘텐츠 신뢰성 기준

- AI 생성 콘텐츠(문서 초안, 일정 추천)는 '참고 자료'로 제공 — 최종 판단은 사용자에게 있음
- AI 문서 추천 근거: 조직 내 과거 문서 벡터 검색 결과 기반 (RAG + ChromaDB)
- AI 일정 추천(AI-02-1)은 캘린더 과거 데이터 기반 — 개인 외부 데이터 미활용

### 4.4 데이터 보존 및 삭제

- 사용자 탈퇴 시 해당 회원이 작성한 문서, 게시글, 업무 데이터는 조직 데이터 자산으로 보존
- 탈퇴 회원의 개인식별 정보는 개인정보보호법 준수를 위해 탈퇴 즉시 익명화 처리
- 익명화 방식: 이름에서 성(姓)만 남기고 나머지 글자를 '@' 기호로 대체 (예: 김철수 → 김@@, 박민 → 박@)
- 사번, 연락처, 이메일은 탈퇴 즉시 완전 삭제, 익명화된 이름만 데이터 귀속 식별자로 유지
- 전자결재 완료 문서는 결재 이력 포함 익명화 상태로 영구 보존

## 5. 게시판 정책

### 5.1 게시판 유형

게시판은 공지사항, 자유게시판 2종으로 고정 운영합니다. 부서별/프로젝트별 게시판 세분화 여부는 추후 협의 후 결정합니다.

| 게시판 유형 | 설명 | 열람 범위 | 작성 권한 | 기능코드 |
| --- | --- | --- | --- | --- |
| 공지사항 | 부서 공식 공지 | 전체 구성원 | 부서 관리자 이상 | Common-04-1~4 |
| 자유 게시판 | 구성원 자유 소통 | 전체 구성원 | 일반 사용자 이상 | Common-04-2~4 |

### 5.2 게시글 작성 기준

- 공지사항 작성: 일반 사용자 이상 작성 가능 (Common-04-2)
- 자유 게시판: 일반 사용자 이상 작성 가능
- 게시글 수정: 본인 작성 글만 수정 가능 (Common-04-3)
- 게시글 삭제: 본인 또는 관리자 삭제 가능 (Common-04-4)

게시글 수정일, 삭제일 요구 데이터 히스토리 관리 필요 → 혼란 야기 방지, 책임 정확

### 5.3 통합 검색 및 필터링

통합 검색(Search-01)은 모든 게시글 대상 키워드 검색을 지원하며, 필터 유형은 세 가지입니다. 확장자 필터(파일 유형별), 카테고리 필터(공지사항 / 자유게시판 2종 고정), 기간 필터(작성일 기준 날짜 범위).

### 5.4 알림 정책

- 결재 알림(Alert-01-3): 결재 요청, 승인, 반려 시 자동 발송. 대리 결재 처리 시 원 결재자에게도 알림 발송
- 채팅 알림(Alert-01-1): 메시지 수신 시 즉시 알림
- 메일 알림(Alert-01-2): 메일 수신 시 즉시 알림
- 업무 추천 알림: AI 캘린더 분석 기반 반복 업무 준비 시작 시점 사전 알림 자동 발송

# 9️⃣ 프로젝트 파일 구성표

| **구분** | **위치** | **파일명/폴더명** | **주요 기능** | **비고** |
| --- | --- | --- | --- | --- |
| **Backend** | `/backend/src/main/java/controller` | `controller/` | REST API 엔드포인트 제어 | Spring Boot 3 |
|  | `/backend/src/main/java/service` | `service/` | 비즈니스 로직 및 AI 연동 서비스 | Java 21 |
|  | `/backend/src/main/java/repository` | `repository/` | 데이터베이스(MariaDB) 접근 로직 | Spring Data JPA |
|  | `/backend/src/main/java/entity` | `entity/` | DB 테이블 및 데이터 구조 정의 | MariaDB 매핑 |
|  | `/backend/src/main/java/config` | `config/` | Security, CORS, Swagger 등 설정 | 환경 설정 |
|  | `/backend/` | `build.gradle` | 의존성 관리 및 빌드 스크립트 | Gradle |
| **Frontend** | `/frontend/src/` | `pages/` | 서비스 주요 화면 및 라우팅 | Vite / JS |
|  | `/frontend/src/` | `components/` | 재사용 가능한 UI 컴포넌트 | 반응형 UI |
|  | `/frontend/src/` | `api/` | 백엔드 API 통신 모듈 관리 | Axios/Fetch |
|  | `/frontend/src/` | `assets/`, `styles/` | 이미지, 아이콘 및 전역 스타일 | CSS/Asset |
|  | `/frontend/` | `vite.config.js` | Vite 빌드 및 프록시 설정 | 개발 환경 설정 |
| **AI Server** | `/ai-server/` | `main.py` | Llama 3 모델 서빙 및 API 노출 | Python / FastAPI |
|  | `/ai-server/` | `models/` | 프롬프트 엔지니어링 및 모델 설정 | Llama 3 8B |
|  | `/ai-server/` | `requirements.txt` | 파이썬 라이브러리 의존성 목록 | Pip 관리 |
| **Database** | `/database/` | `init/` | 초기 DB 스키마 및 더미 데이터 SQL | MariaDB |
| **Infra** | `/Infra/` | `docker-compose.yml` | 멀티 컨테이너 실행 및 네트워크 설정 | Docker 통합 관리 |
|  | `/Infra/` | `Dockerfile` | 각 서비스별 컨테이너 빌드 파일 | 개별 폴더 내 위치 |
| **Docs** | `/docs/` | `design/`, `api-spec/` | Figma 프로토타입 및 API 명세서 | Notion 연동 |
